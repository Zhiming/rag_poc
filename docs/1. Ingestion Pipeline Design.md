# Document Ingestion Pipeline


## 1. Overview

This document describes the design of the document ingestion pipeline for the AI-Powered Security Evaluation Assistant. The pipeline handles two distinct use cases: ingesting free-text evaluation notes and ingesting compliance documents to generate action items. Both pipelines share a common outer shell but diverge in LLM processing, schema, and persistence targets.

| Pipeline | Input | LLM Output | Persistence Target |
|---|---|---|---|
| Evaluation Note | Free-text evaluator note | Structured note with metadata | DynamoDB + OpenSearch |
| Action Item Creation | Compliance doc paragraphs tagged by device type | Flat action item list per device type | DynamoDB only |

---

## 2. Shared Pipeline Shell

Both pipelines share the following outer processing shell before diverging at the LLM processing step.

### 2.1 Submission Contract

Every submission carries the following fields:

- `source_input`: The raw document content (stored directly on the DynamoDB record)
- `pipeline_type`: Enum of `evaluation_note` or `action_item`
- `stable_document_id`: Optional. If present, the submission is treated as an update to an existing record. If absent, the submission is treated as a new document.
- For action item submissions only: paragraphs are pre-tagged by device type by the content creator before submission. The pipeline does not perform device type segmentation.

### 2.2 Text Normalization

Before any hashing or comparison, the raw input text is normalized using the following steps in order:

1. Apply Unicode NFC normalization to resolve encoding differences
2. Convert all characters to lowercase
3. Strip all characters that are not alphanumeric or a space
4. Collapse consecutive whitespace runs to a single space
5. Trim leading and trailing whitespace

This normalization is applied consistently at submission time and when reading existing records for comparison, ensuring that formatting differences and special characters do not produce false mismatches.

### 2.3 Exact Duplicate Detection via Content Hash

After normalization, the pipeline computes a SHA-256 hash of the normalized input. The hash is used for exact duplicate detection:

- If no `stable_document_id` is present in the submission, the pipeline queries DynamoDB for any record where `content_hash` matches and status is `approved` or `pending_review`.
- If a match is found, the submission is rejected early with a descriptive message. The LLM is never invoked.
- If a `stable_document_id` is present, the hash check is skipped entirely. The stable ID signals an intentional update and the pipeline proceeds to locate and overwrite the existing record.

### 2.4 LLM Processing and Schema Validation

The appropriate LangGraph sub-agent is invoked based on `pipeline_type`. The LLM produces structured output conforming to the shared schema library. A validation node runs immediately after the LLM step and checks schema conformance before any write occurs. If validation fails, the pipeline terminates with an error and does not write to DynamoDB. The record is not created.

### 2.5 Human Review Gate

If validation passes, the pipeline writes the structured output to DynamoDB with status `pending_review`. The LangGraph pipeline terminates here. The pipeline does not send the SQS message. That responsibility belongs exclusively to the reviewer approval action triggered through FastAPI.

The reviewer inspects the structured output against the `source_input` stored on the same DynamoDB record. The reviewer either approves or rejects with a written rejection note.

- **Approval**: FastAPI triggers the SQS message send. The consumer Lambda persists data and updates status to `approved`.
- **Rejection**: The rejection note is written to the DynamoDB record and status is updated to `rejected`. The DynamoDB Streams Lambda handles the retry flow.

### 2.6 Retry Flow via DynamoDB Streams

The DynamoDB Streams Lambda is triggered on every status change. When it detects a status of `rejected` it executes the following logic:

1. Read `retry_count` from the record.
2. Use a conditional write to increment `retry_count` only if the current value matches what was read. This guards against at-least-once Streams delivery causing double increments.
3. If `retry_count` is below 3: set status to `pending_processing` and re-invoke the appropriate LangGraph pipeline. The pipeline receives the original `source_input` plus the `rejection_note` appended as additional instruction context to the LLM prompt. The pipeline overwrites the structured output on the same DynamoDB record and resets status to `pending_review`. The human review gate applies again to the new output.
4. If `retry_count` has reached 3: set status to `manual_review_required`. No further pipeline invocation occurs. The record surfaces for human handling outside the automated flow.

## 3. DynamoDB Record Schema

Both pipelines share the following base record fields. Pipeline-specific fields are added on top.

| Field | Type | Description |
|---|---|---|
| `partition_key` | String | Pipeline-specific. See below. |
| `sort_key` | String | Pipeline-specific. See below. |
| `pipeline_type` | Enum | `evaluation_note` or `action_item` |
| `status` | Enum | `pending_processing`, `pending_review`, `approved`, `rejected`, `manual_review_required` |
| `content_hash` | String | SHA-256 of normalized input. Used for exact duplicate detection. |
| `stable_document_id` | String | Caller-provided stable identifier for update routing. |
| `source_input` | String | The raw original submission. Used by reviewer UI and retry Lambda. |
| `structured_output` | JSON | LLM-generated structured document conforming to shared schema. |
| `retry_count` | Integer | Initialized to 0. Incremented on each rejection-triggered retry. |
| `last_retry_at` | Timestamp | Updated on each retry invocation. |
| `rejection_note` | String | Written by reviewer on rejection. Appended to LLM prompt on retry. |
| `created_at` | Timestamp | Submission timestamp. |
| `updated_at` | Timestamp | Last status change timestamp. |

### 3.1 Evaluation Note Key Design

- **Partition key**: evaluator-assigned UUID generated at first submission (e.g. `EVN-2024-0047`)
- **Sort key**: submission timestamp or version counter for tracking revision history

### 3.2 Action Item Key Design

- **Partition key**: compliance document slug derived from document name and version (e.g. `NIST-SP-800-53-Rev5`)
- **Sort key**: device type (e.g. `camera`, `router`, `medical_device`)
- Dedup check operates at the individual record level: before processing, the pipeline checks whether a record already exists for the given partition key and sort key. If it exists and is approved, that device type is skipped while other device types in the same submission continue processing.
- The content hash is stored per record and covers only the paragraphs submitted for that specific device type. A change to camera paragraphs invalidates only the camera record.

## 4. SQS Message Envelope

All SQS messages share a common envelope schema with a `message_type` field and a `schema_version` field. The consumer Lambda uses these fields to route and deserialize the payload correctly. Both the LangGraph node and the consumer Lambda use the same shared schema library, ensuring consistent validation at both ends.

```json
{
  "message_type": "evaluation_note_index" | "action_item_create",
  "schema_version": "1.0",
  "document_id": "<stable_document_id>",
  "pipeline_type": "evaluation_note" | "action_item",
  "structured_output": { ... },
  "submitted_at": "<ISO timestamp>"
}
```

The validation node inside the LangGraph pipeline validates the `structured_output` against the schema before the SQS message is constructed. The consumer Lambda validates again on receipt before any write to DynamoDB or OpenSearch occurs.

# 5. Ingestion Pipeline Design

This section covers the two ingestion pipelines: evaluation note ingestion and action item ingestion. Each pipeline is orchestrated by AWS Step Functions, with discrete Lambda functions handling deterministic preprocessing steps and a LangGraph agent deployed on Bedrock AgentCore Runtime handling LLM transformation. Human review is embedded as a first class state in the state machine using the waitForTaskToken callback pattern.

---

## 5.1 Document Normalization and Hashing

Normalization and hashing run as the first Lambda in both pipelines. These are deterministic operations with no LLM involvement.

### Normalization Pipeline

Normalization applies four transforms in fixed order:

1. Unicode normalization (NFC) must precede lowercasing to avoid encoding-dependent behavior.
2. Lowercasing.
3. Special character stripping must precede whitespace collapsing to avoid compounding spaces where characters were removed.
4. Whitespace collapsing and trimming.

```python
import unicodedata, re, hashlib

def normalize_and_hash(text: str) -> str:
    text = unicodedata.normalize('NFC', text)
    text = text.lower()
    text = re.sub(r'[^a-z0-9 ]', '', text)
    text = re.sub(r' +', ' ', text)
    text = text.strip()
    return hashlib.sha256(text.encode('utf-8')).hexdigest()
```

The SHA-256 digest is computed over the normalized text and written to the Step Functions state payload for use by the exact duplicate check Lambda downstream.

---

## 5.2 Exact Duplicate Check

The exact duplicate check Lambda queries DynamoDB for an existing record matching the content hash computed in 5.1. If a record exists with status `approved` or `pending_review`, the Lambda returns a duplicate signal and Step Functions routes to a terminal state without further processing. If no match is found, execution continues to the next step.

This check is a DynamoDB point lookup by content hash. It is fast, cheap, and requires no embedding model call.

---

## 5.3 Semantic Duplicate Check (Evaluation Note Pipeline Only)

Semantic deduplication runs only in the evaluation note pipeline. It is not applicable to action item ingestion because action items are sourced from structured compliance documents where near-duplicate prose across submissions is not a meaningful concern.

### Flow

1. The Lambda embeds the normalized input text using the same Bedrock Titan embedding model used at query time.
2. A filtered similarity search is run against the OpenSearch evaluation note index, scoped to the same `device_type` and `manufacturer` as the submission.
3. If any existing note exceeds the configured similarity threshold (default 0.92, stored in SSM Parameter Store for live tuning without redeployment), Step Functions routes to the human review state immediately, writing a `pending_review` record with a `possible_duplicate` flag and the matched note IDs attached for reviewer comparison.
4. If no match exceeds the threshold, execution continues to agent invocation.

### Why Device Type Scoped Filtering Matters

Semantic similarity across unrelated device types produces false positives. A camera evaluation note about default credential exposure may score highly against a router note describing the same issue class. Scoping the similarity search to `device_type` and `manufacturer` ensures comparisons are meaningful and reduces reviewer fatigue from spurious duplicate flags.

### Threshold Design

The threshold is intentionally conservative. A false positive (flagging a distinct note as a possible duplicate) is resolved cheaply by the reviewer. A false negative (silently passing a note that duplicates existing institutional knowledge) results in index pollution and degraded retrieval quality. The threshold is stored in SSM Parameter Store rather than hardcoded so it can be tuned as real evaluation data accumulates without a pipeline redeployment.

## 5.4 Agent Design: Evaluation Note Pipeline

The evaluation note agent is deployed on Bedrock AgentCore Runtime. It is invoked by a thin invoker Lambda that calls `InvokeAgentRuntime` via boto3, passing the normalized text as the payload. The agent has a single responsibility: transform free text evaluation notes into structured documents.

### LangGraph Node Definitions

| Node | Responsibility |
|---|---|
| `invoke_llm` | Call the Bedrock LLM with the normalized input text and a structured prompt requesting a sectioned output covering Findings, Remediation, Compliance Impact, and Summary. If `rejection_note` is present in state from a prior retry, append it as additional instruction context. Write `structured_output` to state. |
| `validate_schema` | Validate `structured_output` against the evaluation note schema. If invalid, write `validation_errors` to state and route back to `invoke_llm` with the errors appended to the prompt. If max retries exceeded, raise an error to Step Functions. |

### Node Connections

```
invoke_llm
  --> validate_schema
        --> [valid]              --> return structured output to Step Functions
        --> [invalid, retrying]  --> invoke_llm (with validation errors appended)
        --> [max retries exceeded] --> raise error --> Step Functions error terminal
```

### Agent State

```python
class EvaluationNoteAgentState(TypedDict):
    normalized_text: str
    structured_output: Optional[dict]
    validation_errors: Optional[List[str]]
    rejection_note: Optional[str]
    retry_count: int
```

## 5.5 Agent Design: Action Item Pipeline

The action item agent is deployed on Bedrock AgentCore Runtime and invoked by the same invoker Lambda pattern. Its single responsibility is to parse structured compliance document input and produce a list of action items per device type.

### LangGraph Node Definitions

| Node | Responsibility |
|---|---|
| `invoke_llm` | Call the Bedrock LLM with the input paragraphs for each device type and a prompt requesting structured action item output. If `rejection_note` is present, append it. Write `structured_output` to state. |
| `validate_schema` | Validate `structured_output` against the action item schema. If invalid, write `validation_errors` to state and route back to `invoke_llm`. If max retries exceeded, raise an error to Step Functions. |

### Node Connections

```
invoke_llm
  --> validate_schema
        --> [valid]               --> return structured output to Step Functions
        --> [invalid, retrying]   --> invoke_llm (with validation errors appended)
        --> [max retries exceeded] --> raise error --> Step Functions error terminal
```

```python
class ActionItemAgentState(TypedDict):

```

## 5.6 Step Functions Orchestration

Both pipelines share a single Step Functions state machine. The state machine branches on `pipeline_type` at the start to route to the correct path.

### Evaluation Note Path

```
Normalize + Hash Lambda
  --> Exact Duplicate Check Lambda
        --> [duplicate found] --> END (status: duplicate_rejected)
        --> [no duplicate]   --> Semantic Duplicate Check Lambda
                                    --> [possible duplicate] --> Write DynamoDB (pending_review, possible_duplicate flag) --> waitForTaskToken
                                    --> [no duplicate]       --> Invoke Agent Lambda (evaluation note agent)
                                                                    --> [agent error] --> END (status: agent_failed)
                                                                    --> [agent success] --> Write DynamoDB Lambda (status: pending_review)
                                                                                              --> waitForTaskToken

waitForTaskToken
  --> [approved] --> Write DynamoDB Lambda (status: approved) --> Index to OpenSearch Lambda --> END
  --> [rejected] --> Write DynamoDB Lambda (status: rejected) --> END
```

### Action Item Path

```
Normalize + Hash Lambda
  --> Exact Duplicate Check Lambda
        --> [duplicate found] --> END (status: duplicate_rejected)
        --> [no duplicate]   --> Invoke Agent Lambda (action item agent)
                                    --> [agent error] --> END (status: agent_failed)
                                    --> [agent success] --> Write DynamoDB Lambda (status: pending_review)
                                                              --> waitForTaskToken

waitForTaskToken
  --> [approved] --> Write DynamoDB Lambda (status: approved) --> END
  --> [rejected] --> Write DynamoDB Lambda (status: rejected) --> END
```

### Human Review via waitForTaskToken

When the state machine reaches the `waitForTaskToken` state, Step Functions generates a unique task token for that execution and pauses. The Write DynamoDB Lambda stores the task token alongside the `pending_review` record. The reviewer notification (via SES or SNS) includes a deep link to the React review UI with the task token embedded.

When the reviewer approves or rejects through the React frontend, the FastAPI backend receives the decision and calls `StepFunctions.SendTaskSuccess` (approve) or `StepFunctions.SendTaskFailure` (reject) with the task token. Step Functions resumes execution and routes to the appropriate terminal state.

This means the complete lifecycle of a document, from submission through preprocessing, human review, and final persistence, is a single traceable Step Functions execution with timestamps and status at every state transition.

## 5.7 OpenSearch Index Design

This section covers the index schema and deduplication query behavior at ingestion time. Retrieval strategy, hybrid search, and MMR reranking are covered separately in the retrieval design section.

A single index holds one document per evaluation note. The content fields are split by section so that each embedding is computed over semantically coherent text rather than a mixed document. The findings embedding drives retrieval relevance: evaluator queries describe what they are currently observing and are matched against `findings_embedding`. The remediation embedding drives MMR diversity: after the findings query returns a candidate pool, MMR is applied over `remediation_embedding` across those candidates to select a diverse set of remediation approaches. Both embeddings and both text fields are returned in a single query response, so no second lookup is needed.

### Index: `evaluation_notes`

| Field | Type | Purpose |
|---|---|---|
| `document_id` | keyword | DynamoDB partition key. Used to fetch the canonical record for editing and review. |
| `findings_text` | text (BM25) | Findings section text. Indexed for BM25 lexical search. Source text for `findings_embedding`. |
| `remediation_text` | text | Remediation section text. Not queried directly. Returned in the response payload as context for the LLM. Source text for `remediation_embedding`. |
| `findings_embedding` | knn_vector (1536 dims) | Dense vector computed over `findings_text`. Used for semantic similarity search at query time and for the deduplication query at ingestion time. Stored with `store: true`. |
| `remediation_embedding` | knn_vector (1536 dims) | Dense vector computed over `remediation_text`. Used by MMR at query time to select diverse remediations across the matched candidate pool. Stored with `store: true`. |
| `device_type` | keyword | Pre-filter in deduplication queries and retrieval queries. |
| `manufacturer` | keyword | Pre-filter in deduplication queries and retrieval queries. |
| `note_type` | keyword | `incident_report`, `compliance_audit`, `network_scan`, `firmware_review`. |
| `has_incident` | boolean | Filter for incident-scoped retrieval queries. |
| `evaluation_date` | date | Recency sorting within result sets. |
| `facility_id` | keyword | Scoping filter for facility-specific queries. |
| `compliance_refs` | keyword[] | Compliance framework identifiers referenced in the note. |

### Deduplication Query Behavior

The semantic deduplication Lambda runs a pure `knn_vector` search against `findings_embedding`, scoped by `device_type` and `manufacturer` pre-filters. No hybrid search or MMR is applied at ingestion time. The goal is to find the single most similar existing finding, not a diverse result set. If any candidate exceeds the configured similarity threshold, the pipeline flags the submission as a possible duplicate and routes to human review.

## 6. Downstream Consumer Lambda

The consumer Lambda subscribes to the SQS queue and processes messages based on `message_type`.

- For `evaluation_note_index` messages: persist the structured note to DynamoDB (update status to `approved`), then write document chunks to OpenSearch. Chunk boundaries follow the parent-child chunking strategy defined separately. The DynamoDB Streams Lambda handles any subsequent update-triggered re-indexing.
- For `action_item_create` messages: persist the structured action item list to DynamoDB (update status to `approved`). No OpenSearch write occurs for action items since they are retrieved by direct DynamoDB lookup by compliance document slug and device type.
- The consumer Lambda validates the `structured_output` against the shared schema library on receipt before any write occurs. If validation fails, the message is sent to the SQS dead letter queue rather than retried indefinitely.
- OpenSearch writes only occur on explicit approval via SQS message. No partial writes occur during the LangGraph retry cycle.
